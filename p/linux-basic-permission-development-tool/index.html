  • 如何退出vim？

    直接输入vim时使用 shift+冒号+q退出，

    写了代码后 shift+冒号+wq，如果是插入模式下需要先按esc 退回到命令模式

    1. 如果你没有做任何修改，只是查看了文件并想退出：
      • 确保你处于命令模式（按 Esc 键可以确保你处于命令模式）。
      • 输入 :q 然后按 Enter 键。这将退出 Vim，且不会保存任何修改。
    2. 如果你做了修改并希望保存更改然后退出：
      • 确保你处于命令模式。
      • 输入 :wq 然后按 Enter 键。这将写入（保存）文件并退出 Vim。
      • 或者，你可以先输入 :w 然后按 Enter 键来保存文件，之后输入 :q 再按 Enter 键来退出。
    3. 如果你做了修改但不希望保存更改并想退出：
      • 确保你处于命令模式。
      • 输入 :q! 然后按 Enter 键。这将强制退出 Vim，且不保存任何修改。
    4. 如果你处于插入模式（正在输入文本），并且想退出：
      • 首先按 Esc 键回到命令模式。
      • 然后按照上述步骤之一退出 Vim。

3.编译器gcc/g++

3.1 程序翻译 — gcc/g++编译选项

gcc和g++都叫做编译器，主要工作就是将我们写的C/C++代码翻译成可执行二进制程序

  • gcc只用来编译C
  • g++编译C/C++

g++ code.cpp -std=c++11

  1. 预处理（进行宏替换/去注释/条件编译/头文件展开等) ，还是C语言

    -E表示从现在开始，进行程序的翻译，一旦预处理做完，就停下来

    如果不带-o，会把预处理的结果打印到显示器上。

    所以code.i保存的就是code.c预处理后的结果

    • -o 表明-o后面的文件名称
    • -E 表示从现在开始，进行程序的翻译，一旦预处理做完，就停下来

    头文件展开：将要包含的头文件相关内容拷贝到源文件

    ​ 因为头文件里可能还包含其它头文件，所以展开类似于递归式展开

    ​ 先追到根上的头文件，然后依次拷贝，最后将需要的头文件内容

    ​ 拷贝到可执行程序里，最终编译只编译源文件

    Linux中C语言标准头文件在该路径下/usr/include/

    函数的实现在库里

  2. 编译（C语言–>汇编)

    gcc -S code.i -o code.s：-S表示开始进行程序翻译，编译做完，形成汇编，就停下来

  3. 汇编（生成机器可识别代码即汇编—>二进制）

    gcc -c code.s -o code.o可重定位目标二进制文件，无法执行

    -c表示开始进行程序的翻译，编译做完，形成汇编，就停下来

  4. 连接（生成可执行文件或库文件)

    将我们的.o文件和库文件进行连接，形成可执行文件

总结：

1）-E -S -c 预处理，编译，汇编

2）.i .s .o 一些镜像文件的后缀

3.2 为什么要编译，为什么汇编、编译器自举，理解条件编译

为什么要编译，汇编

  1. 早期：纸带打孔的方式编程 – 二进制编程

  2. 汇编语言 — 编译器就要诞生了

  3. 面向过程，面向对象的语言

为什么需要把我们用其他语言写的代码转成汇编？

**减少语言开发的成本。**因为汇编语言出现的比较早，已经有编译器可以将汇编语言翻译成

二进制语言，所以我们只需要考虑把C/C++等语言翻译成汇编语言。

编译器自举

先有语言，还是先有编译器？

  • 先有编译器，二进制的编译器，能编译汇编语言所写的软件，编译器自己是软件

  • 我们可以用汇编语言重构编译器，这个编译器软件先经过二进制编译器，然后形成

    用汇编写的编译器。

  • 此后，我们就不需要使用二进制编译器了，而是可以使用用汇编写的编译器

条件编译

版本管理，通过条件编译形成不同版本

3.4动静态库和静态链接—理论部分

动态库是共享的，所以不能丢失，一旦丢失，所有依赖动态库都会运行出错

**静态链接：**把我们要的库方法实现，直接拷贝到我们的可执行程序中

​ 一旦形成可执行程序，即使把静态库删了，程序照样能跑

总结：

动静态链接，首先系统里要有动态库，静态库，都有的情况下就可以对程序进行链接

如果链接的是动态库，就是动态链接；如果链接的是静态库，就是静态链接

静态链接优点：不依赖库

缺点：浪费空间（磁盘和内存空间）

4.4.1 第一部分-依赖关系

  • 冒号左边，比如上图中的clean和mytest都是一个目标文件

  • 冒号右边，比如上图中的test.c是依赖文件，如果依赖文件有很多个，那么叫依赖文件列表

makefile怎么决定源文件到底要不要重新编译的呢？

  1. 先有源文件，然后gcc形成可执行程序。
  2. 之后再修改源文件，那么源文件的修改时间就比可执行程序的时间新
  3. 再次编译时，发现可执行程序的生成时间比源文件的修改时间晚，说明这是旧的可执行程序
  4. 所以这时允许重新编译源文件

结论：通过对比源文件和可执行文件的修改时间（Modify时间）

4.1.2 文件的ACM时间

文件=内容+属性

可以通过stat 文件名的方式查看文件的ACM时间

  • 文件的内容修改，Modify时间变

  • 文件的属性修改，Change时间变

    • 修改文件的内容后，Modify时间变了，同时，文件的大小也变了，所以Change时间也会变
    • 如果只修改文件的权限，那么就只有Change时间会变

  • Access是文件的最新访问时间，cat一个文件后，就会变。

    但是不是每次访问后，这个时间都会变

touch一个文件后，这个文件的ACM时间都更新

makefile通过对比源文件和可执行文件的修改时间（Modify时间）来决定源文件

到底要不要重新编译的。

所以只要我们touch一下源文件，那么源文件的M时间一定比可执行程序的时间新

这时再make就又可以编译了

所以.PHONY修饰一个符号后，总是被执行的原因就是忽略对比时间，下面这条命令总是执行
